  W-02: reset = location.reload() IN batch-x MODULES
    brutalist_massing and triangulated_signal_mesh reset via location.reload().
    Under HTTP server: works but causes full iframe reload (slow, jarring).
    Under file://: causes full file reload (no state persistence anyway).
    Blocks "soft reset" concept (reset to initial params without reload).
    FIX DIRECTION: Add a named reset function in batch-x modules that reinitializes
    state without reload. Expose via id="reset" or standalone_common handler.

  W-03: standalone_common.js PATH DEPENDENCY
    All 4 A01 modules load "../standalone_common.js".

9. SHELL-COMPLIANCE RISKS
────────────────────────────────────────────────────────────────────────────────

  SC-01: HAND-AUTHORED MODULES IN MAIN LAB DO NOT HAVE CONSISTENT IDs
    Verified modules (gyroid, sdf_raymarch, displacement, etc.) use mix of
    Template A and Template C IDs. The unified shell bridge fallback list
    covers all known sets. But each new hand-authored module could introduce
    new or missing IDs that break bridge routing.

  SC-02: GEOMETRY_ONLY_MODE VARIANT (body.geoOnly vs always-on CSS)
    Generated modules use body.geoOnly class (toggle).
    Some older hand-authored modules may use always-on GEOMETRY_ONLY_MODE CSS.
    The unified shell's revealModulePanel injection targets both broadly.
    Worst case: a module with always-on GEOMETRY_ONLY_MODE CSS will have
    body{position:static} injected permanently — which may break its layout.

  SC-03: CSS ISOLATION BETWEEN SHELL AND IFRAME CONTENT
    The unified shell's overlay CSS (.previewOverlay, .ctrlBtn overrides)
    is injected into the module iframe's <head>.
    ctrlBtn class is ALSO used inside modules. This could cause style leakage
    if modules have their own .ctrlBtn rules that conflict with the injected ones.

  SC-04: ANIMATION LOOP WHEN SHELL IS NOT VISIBLE
    Modules that run requestAnimationFrame loops continue running even when
    the module is hidden (another module is active in the shell).
    In the unified shell, the previous iframe's src is replaced by new src
    (frame.src = module.file) — so the old document is unloaded. OK.
    But during the brief period while the old frame is still loaded before
    src change, its loop continues. Low impact.

  SC-05: DRAG-TO-RESIZE / INTERACTIVE CANVAS BEHAVIORS
    Some modules may capture mousedown/touchstart on the canvas and prevent
    propagation. Under iframe, this is contained. But keyboard shortcuts
    captured by the module frame (event.stopPropagation) will not bubble to
    the shell. Shell keyboard shortcuts (/search, arrows, S, T, etc.) depend
    on document-level keydown in the shell — these work ONLY when the shell
    document has focus, not when the module iframe has focus.
    This is a known limitation and is acceptable behavior.

11. PROCESSOR-LOAD RISKS
────────────────────────────────────────────────────────────────────────────────

  PL-01: MULTIPLE requestAnimationFrame LOOPS
    Each module runs its own rAF loop. When switching modules in the shell,
    the old frame.src is replaced — browser terminates old document and its
    loops. This is handled by the browser automatically.
    LOW RISK under normal module switching.

  PL-02: PHYSICS CANVAS TRANSFORM + filter IN batch-x
    experimental-physics in batch_15 applies CSS transform + filter to canvas.
    On every tick: translate3d, scale, and contrast filter.
    CSS filter on canvas forces GPU compositing layer — can spike GPU usage.
    In A01 context (single iframe) this is acceptable.
    In unified shell with many module switches, GPU layer may linger briefly.
    LOW-MEDIUM RISK.

  PL-03: PLACEHOLDER THUMBNAILS REPLACED BY TEXT
    In A01, when modules have no thumb, the fallback is a .moduleLabel span
    with the module title. This is text rendering, not image load.
    NO performance impact — actually cheaper than thumbnails.

  PL-04: PARALLEL FETCH FOR MANIFEST (unified only)
    Unified shell fetches manifest on boot. One network request.
    Under HTTP server, this is near-instant for a local JSON file.
    NO performance concern.

  PL-05: p5.js CDN LOAD TIME
    p5.min.js is ~800KB. Loading from CDN adds first-load latency.
    Under local HTTP + cache, subsequent loads are fast.
    MEDIUM RISK for first-open experience.

12. RECOMMENDED SHARED CONTRACT (MINIMAL USEFUL VERSION)
────────────────────────────────────────────────────────────────────────────────
  Do NOT implement yet. This is a design recommendation only.

  MINIMUM VIABLE MODULE CONTRACT:
  ─────────────────────────────────
  Every shell-compatible module should either:
    (A) Have these canonical button IDs in its DOM, OR
    (B) Expose them via standalone_common.js aliases in install()

  Canonical IDs (in priority order):
    pause       ← toggles animation loop on/off
    reset       ← returns to initial params (soft, no reload preferred)
    newSeed     ← generates a new random seed string
    randomAll   ← randomizes all unlocked parameters
    save        ← downloads canvas as PNG at 1x
    save2x      ← downloads canvas as PNG at 2x
    thumb       ← downloads canvas as small PNG thumbnail
    btnUndo     ← reverts to previous param state (if history exists)
    btnRedo     ← re-applies forward param state

  OPTIONAL (capability-declared, gracefully absent):
    randomForm  ← randomizes form/shape params only
    randomColor ← randomizes color/palette params only
    rec         ← starts/stops video recording
    btnTransparentBg ← toggles transparent background mode
    copyP       ← copies current param state to clipboard
    pasteP      ← pastes param state from clipboard

  CAPABILITY FLAG (future, in manifest or module metadata):
    capabilities: ["save", "thumb", "undo", "redo", "rec"]
    Shell reads capabilities → shows only supported buttons.
    Unsupported buttons are hidden (not shown as disabled).

  THE SMALLEST USEFUL STEP:
    Require only:
      1. Either canonical ID in DOM, or standalone_common.js alias
      2. save / reset / pause as mandatory minimum (3 IDs)
      3. All others optional with graceful status message
    This matches current A01 behavior and requires zero new code.

  WHAT NOT TO DO NOW:
    Do not build a formal JSON capability schema yet.
    Do not add capability checking to shells yet.
    Do not require modules to emit capability declarations.
    Prove the pattern works across 4 A01 modules + 2-3 promoted escrow modules,
    then formalize.
