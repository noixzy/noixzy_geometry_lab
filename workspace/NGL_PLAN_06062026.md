# Architectural Audit: noixzy_generative_lab

## Summary
The project currently has **47 gallery-listed live modules**: **12 hand-authored** and **35 generated**. The generator also still defines **`sdf`**, making **36 generated definitions** in `build_lab.js`; `sdf/noixzy_sdf.html` exists, but it is not currently listed in the gallery.

A unified shell is feasible **without reducing functionality** if it uses an iframe-hosted architecture and a thin module adapter/message bridge. The shell should not merge render engines. It should centralize navigation, toolbar, schemas, presets, randomization, exports, audio mapping, transparency, thumbnail capture, and metadata while preserving standalone module files.

## Feature Inventory

**Shared By All Or Nearly All Modules**
- Standalone HTML module per directory, mostly p5-based and double-clickable.
- Gallery with cards, thumbnails, search, metadata, and hand-authored/generated badges.
- Canvas/stage rendering, seed/state concepts, theme or palette controls, favorites/pins, thumbnail save, transparent background toggle, bottom toolbar, side/thumb module navigation, preview history UI, localStorage persistence.
- Common production exports exist broadly: PNG, 2x PNG in many modules, thumbnail capture, and video/PBR in generated modules.

**Hand-Authored Only Or Stronger In Hand-Authored**
- Custom render architectures for flagship/3D-ish modules: `grid_extrude`, `sdf_raymarch`, `gyroid`, `displacement`, `displacement_primitives`, `mandelbulb`, `fold`, `metafluid`, `hex_grid`, `rose_curve`, `lissajous_mesh`, `torus_knot`.
- Rich per-module bespoke control models, starter/local presets in some modules, unique state capture/apply logic, custom shader/raymarch/SDF-style render paths.
- `grid_extrude` is closest to generated modules structurally; `sdf_raymarch`, `gyroid`, and `torus_knot` diverge more strongly.

**Generated Only Or Stronger In Generated**
- Generated modules are stamped from `build_lab.js` with one shared engine and per-piece `system` definitions plus `render()` and often `heightField()`.
- Consistent groups: `system`, `extrude`, `material`, `frame`, `look`, `motion`.
- Shared controls include random all/form/color, copy/paste params, audio band mapping, transparent background, thumb save, video, save PNG, save 2x, PBR maps, SVG hook where available, drag pan, wheel zoom, preview history, nav rail.
- Generated family includes the gallery-listed modules from `moire_field` through `stipple`; `sdf` exists as generated but is not gallery-listed.

**Experimental, Partial, Or Unverified**
- Audio-reactive controls exist across generated and many hand-authored modules but are called out in project notes as unverified backlog.
- No module currently exposes a postMessage bridge; unified shell control is not implemented.
- Hand-authored modules have uneven support for video, PBR maps, 2x export, copy/paste, local presets, and exact theme API.
- `noixzy_lab_shell_v1.html` is a static prototype, not a real host.
- Gallery metadata is duplicated between `gallery/index.html` and `build_lab.js`; `sdf` mismatch is a concrete metadata drift risk.

## Architecture Comparison

**Generated Module Pattern**
- Source of truth: `build_lab.js`.
- One shared HTML/CSS/UI/runtime shell generated around many per-piece definitions.
- Parameters are declarative objects with keys, group, label, range, default, and rebuild/randomization hints.
- Render contract is stable: `build()`, `render(g,pal)`, optional `heightField(G)`.
- State contract is mostly `seed`, `P`, `colorState`, `activeTheme`, localStorage favorites.

**Hand-Authored Pattern**
- Source of truth: individual HTML files.
- Similar visible UI language but inconsistent internal APIs.
- Some use `PARAMS` and `P`; others use bespoke arrays, `P_CONTROLS`, direct DOM IDs, custom state objects, starter presets, or raymarch-specific color controls.
- Render loops and export paths are module-specific.
- Feature parity is visual, not architectural.

**Core Difference**
Generated modules already have a schema-driven engine. Hand-authored modules have equivalent user-facing concepts but need adapters to expose them through the same shell protocol.

## Unified Shell Recommendation

Build a unified shell as an **iframe host plus adapter protocol**, not a merged monolith.

**Toolbar**
- Shell-owned toolbar sends actions: `newSeed`, `reset`, `pause`, `savePng`, `save2x`, `recordVideo`, `saveThumb`, `pbrMaps`, `pin`, `copy`, `paste`, `transparentBg`, `audioToggle`.
- Modules may advertise unsupported actions; shell disables or soft-hides unavailable buttons.

**Module Navigation**
- Shell reads one canonical module manifest containing id, title, type, path, thumbnail, tags, description, supported capabilities.
- Left rail loads `../<id>/noixzy_<id>.html` into an iframe.
- Keep direct module pages standalone.

**Control Panel And Parameter Schema**
- Define a canonical schema: `{key, label, group, type, min, max, step, default, options, randomizable, rebuild, audioMappable}`.
- Generated modules can expose schema directly from `PARAMS`.
- Hand-authored modules get per-module adapters mapping shell keys to native state/DOM/state functions.

**Preset System**
- Shell presets should store `{moduleId, seed, params, theme, colors, transparentBg, audioMap, createdAt, version}`.
- Preserve existing module-local favorites/presets during migration; shell presets are additive.

**Randomization**
- Shell supports `randomAll`, `randomForm`, `randomColor`, delegated to module if native; otherwise use schema ranges.
- Respect `randomizable:false` and rebuild flags.

**Export Pipeline**
- Shell delegates exports to active iframe first.
- Standard actions: PNG, 2x PNG, thumbnail, video, PBR maps, optional SVG.
- Long term, module replies with capability/status events so the shell can show progress/toasts.

**Audio-Reactive Controls**
- Prefer shell-owned audio input/analyser in the long term.
- Send normalized bands `{bass, mid, high, presence}` to the module, plus `audioMap`.
- Short term, keep existing module-local audio panels and add bridge compatibility.

**Transparent Background**
- Standardize as `setTransparentBg(bool)`.
- Generated modules already have the concept; hand-authored adapters should map to existing toggles/state.

**Thumbnail Generation**
- Shell action asks module to render/capture a 16:9 thumbnail and returns/downloads it.
- Gallery thumbnails remain files under `gallery/thumbs/`, with a future manifest update step.

**Gallery Metadata**
- Replace duplicated arrays with one generated/static manifest consumed by gallery, shell, and module nav.
- Include `kind: "hand-authored" | "generated"`, capabilities, status, and thumbnail path.

## Blockers, Risks, And Migration Effort

**Blockers**
- No current postMessage bridge.
- Hand-authored modules lack a common schema/state API.
- Gallery and generator metadata are already drifting: generated `sdf` exists but is not gallery-listed.
- Some features are present in UI but not verified, especially audio/video/export variants.

**Compatibility Risks**
- Editing generated modules directly would be lost; generator must be changed first.
- Hand-authored modules may have custom IDs/state names, so a naive shell parameter panel would fail.
- Running inside iframes can affect download prompts, clipboard, audio/mic permission, keyboard shortcuts, and File System Access thumbnail saving.
- Theme/palette semantics differ: visual “theme” and render “palette” are sometimes coupled, sometimes bespoke.

**Migration Effort**
- Generated bridge: low to moderate, mostly one edit in `build_lab.js` plus regeneration.
- `grid_extrude`: moderate, close to generated schema.
- Other hand-authored modules: moderate to high, adapter-by-adapter.
- Unified shell: moderate, because prototype exists but must replace its fake stage with real iframe/module protocol.
- Verification: high enough to require a matrix across module types, actions, iframe mode, and standalone mode.

## Phased Implementation Plan

**Phase 1: Canonical Inventory And Manifest**
- Create one module manifest covering all live modules and resolving whether `sdf` should be visible.
- Add capability flags for actions, exports, audio, transparent background, presets, PBR, SVG, and schema availability.
- Update gallery and future shell to consume this manifest.

**Phase 2: Module Bridge Contract**
- Add a generated-module bridge in `build_lab.js`: `getSchema`, `getState`, `setParam`, `setSeed`, `setTheme`, `setTransparentBg`, `action`, `audioBands`.
- Emit state/capability messages from modules to parent shell.
- Regenerate generated modules only from the generator.

**Phase 3: Hand-Authored Adapters**
- Start with `grid_extrude`, then one hard case such as `sdf_raymarch`, then roll through the remaining hand-authored modules.
- Each adapter exposes canonical schema/state/actions while preserving native UI and standalone behavior.
- Do not force hand-authored internals into the generated engine.

**Phase 4: Real Unified Shell**
- Convert `noixzy_lab_shell_v1.html` into `shell/index.html`.
- Use iframe stage, manifest-driven rail, shell-owned toolbar/control panel/readout/theme picker, and debounced postMessage controls.
- Disable unsupported actions based on module capability replies.

**Phase 5: Presets, Randomization, And Exports**
- Implement shell presets and import/export JSON.
- Route randomization through native module handlers where available, schema fallback otherwise.
- Standardize PNG, 2x, thumbnail, video, PBR, and optional SVG action responses/status.

**Phase 6: Audio And Transparency Verification**
- Decide whether audio remains module-local for v1 or becomes shell-owned.
- Standardize audio maps and transparent background behavior across generated and hand-authored modules.
- Verify mic/file audio permissions in iframe and standalone contexts.

**Phase 7: Regression Matrix And Lock**
- Test at least: one generated heightfield module, one generated non-heightfield module, `grid_extrude`, `sdf_raymarch`, one shader/SDF hand-authored module, and one curve hand-authored module.
- Acceptance criteria: shell control works, standalone modules still work, gallery opens every card, thumbnails exist, console is clean, exports download, audio/transparent modes do not regress.
